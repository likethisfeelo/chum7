# Cheer & Thanks Phase 2 라이브 레디니스 실행안

## 0) 현재 전제
- Phase 1의 코어(상호작용 API, 통계 materializer, 관측 대시보드, 운영 문서)는 이미 가동 가능한 수준까지 확보.
- Phase 2는 “기능 추가”보다 **예약 발송 신뢰성 + 운영 자동화 + 장애 복구성**을 라이브 기준으로 끌어올리는 단계.
- 초기 라이브 트래픽은 **MAU 1만 미만(초기 활성 1천~3천, 피크 동시접속 수백 명 수준)** 을 기준으로 설계한다.

---

## 1) Phase 2 목표 (Live 가능한 상태 정의)

### 제품/기능 목표
1. 예약 응원이 지정 시점에 안정적으로 발송된다.
2. 실패 건은 자동 재시도 후 dead-letter로 격리되며 운영자가 추적·재처리할 수 있다.
3. 취소·수정·노출 정책이 예측 가능하게 고정되어 CS 이슈가 줄어든다.

### 운영/신뢰성 목표
- 예약 발송 성공률(초기 라이브): **99.0%+**
- 재시도 후 dead-letter 비율(초기 라이브): **1.0% 미만**
- p95 API 지연(조회 계열): **700ms 이내**
- 장애 감지→1차 조치 시작: **30분 이내**

### 스케일 업그레이드 트리거(1만명 근접 시)
- 최근 2주 연속 MAU 8천 이상 또는 예약 응원 일 처리량 5만 건 이상이면 상향 목표로 전환.
- 상향 목표:
  - 예약 발송 성공률: **99.5%+**
  - 재시도 후 dead-letter 비율: **0.5% 미만**
  - p95 API 지연: **500ms 이내**
  - 장애 감지→1차 조치 시작: **15분 이내**

---

## 2) Phase 2 핵심 스코프

## Track A. 예약 발송 신뢰성 고도화 (필수)
- Cheer 레코드에 `retryCount`, `nextRetryAt`, `failureCode`, `deadLetterReason`, `sentAt` 표준화.
- 재시도 정책 고정: 최대 3회(1m/5m/15m).
- 재시도 초과 건은 `chme-{stage}-cheer-dead-letters`로 이동.
- DLQ 적재 시 원본 스케줄 시각과 마지막 오류 코드 보존.

**완료 기준**
- 실패 주입 테스트에서 재시도/격리 상태 전이 100% 일치.
- DLQ 샘플 30건 수동 검증 시 필수 필드 누락 0건.

## Track B. 예약 취소/경합 정책 고정 (필수)
- 취소 cut-off: 발송 5분 전(예외 없음).
- 취소 API와 스케줄러 경합 시 우선순위 규칙 문서화:
  - cut-off 이전 취소 확정 시 발송 금지.
  - cut-off 이후는 발송 프로세스 우선.
- 에러 코드 표준화: `CANCELLATION_WINDOW_CLOSED`, `ALREADY_SENT`, `ALREADY_CANCELLED`.

**완료 기준**
- 경합 시뮬레이션 3케이스(조기취소/경계시각/지연취소) 모두 기대 응답 확인.

## Track C. 운영 콘솔/런북 강화 (필수)
- Admin Docs에 아래 운영 액션 추가:
  - dead-letter 목록 API: `GET /admin/cheer/dead-letters`
  - dead-letter 상세 API: `GET /admin/cheer/dead-letters/{cheerId}`
  - dead-letter 지표 API: `GET /admin/cheer/dead-letters/stats`
  - 단건 재처리 API: `POST /admin/cheer/dead-letters/{cheerId}/requeue`
  - 배치 재처리 API: `POST /admin/cheer/dead-letters/requeue-batch` (최대 50건)
  - 조건 재처리 API: `POST /admin/cheer/dead-letters/requeue-by-query` (기간/failureCode 기반, dry-run 지원)
  - 재처리 금지 기준(수신자 탈퇴, 정책 만료 등)
- 알람 체계:
  - dead-letter 건수 급증
  - 예약 발송 실패율 급증
  - 스케줄러 실행 누락
- 온콜 체크리스트(10분/30분/24시간 액션) 템플릿 확정.
- 초기 라이브(소규모) 기준으로 **평일 온콜 + 주말 콜백 체계** 운영, 야간 상시대기는 Phase 2.5에서 재평가.

**완료 기준**
- 모의 장애 드릴 2회에서 런북만으로 복구 절차 수행 가능.

## Track D. API 품질·스케일 점검 (권장)
- `/cheers/scheduled`, `/cheers/stats` 페이징/필터 파라미터 표준화.
- 조회 핫패스 인덱스/쿼리 패턴 점검(챌린지 기간 필터 중심).
- 대량 조회 시 source mix(bucket/realtime) 비율 모니터링.

**완료 기준**
- 대표 시나리오 부하 테스트에서 p95 700ms 이내.
- 초기 트래픽(예: 1분당 300~500 요청) 구간에서 에러율 1% 미만 유지.

## 3) 4주 실행 로드맵 (권장)

### Week 1 — 계약 고정 & 데이터 모델 반영
- 상태머신/에러코드/취소정책 확정(ADR 업데이트).
- DynamoDB 스키마/인덱스/TTL 검토.
- DLQ 테이블 및 기본 IAM 권한 반영.

### Week 2 — 재시도/DLQ 구현 + 단위 테스트
- 스케줄 발송 Lambda에 백오프/재시도 적용.
- DLQ 적재/조회 기본 API(관리자 전용) 구현.
- 실패 코드 매핑(네트워크/권한/검증/내부오류) 정리.

### Week 3 — 운영 자동화 & 관측
- CloudWatch 지표/알람/대시보드 위젯 추가.
- 재처리 스크립트/명령 템플릿 정리.
- 모의 장애 드릴(발송 장애, 부분 장애).

### Week 4 — 스테이징 고정 검증 & 점진 배포
- 스테이징 soak test(최소 48시간).
- Feature flag 단계 배포(내부 100명 → 10% → 30% → 100%).
- go/no-go 리뷰 후 라이브 오픈.

---

## 4) Go-Live Gate (반드시 모두 통과)
1. **기능 게이트**: 예약 생성/취소/발송/재시도/DLQ/재처리 E2E 녹색.
2. **신뢰성 게이트**: 실패 주입 시 데이터 유실 0건.
3. **관측 게이트**: 알람 1차 대응자가 대시보드만으로 원인 분류 가능.
4. **운영 게이트**: 온콜 문서 기준 신규 운영자도 30분 내 대응 플로우 수행.
5. **릴리즈 게이트**: 롤백 커맨드/조건 사전 검증 완료.

### 초기 라이브(1만 미만) 운영 완화 기준
- P1/P2 알람은 근무시간 내 대응 원칙, P0만 즉시 대응.
- 수동 재처리 허용량: 일 최대 200건(초과 시 자동 배치 재처리 우선).
- SLA 공지는 “베타 운영 기준”으로 명시하고, 정식 SLA는 스케일 업그레이드 트리거 충족 후 확정.

---

## 5) 리스크와 선제 대응
- **발송 시각 민감도 리스크**: 재시도 지연이 의미를 약화.
  - 대응: 최대 재시도 횟수 제한 + dead-letter 격리 후 사후 보정.
- **운영 피로도 리스크**: 실패 건 증가 시 수동 대응 과부하.
  - 대응: 실패 코드 분류 자동화 + 우선순위 큐.
- **정책 혼선 리스크**: 취소/발송 경합 문의 증가.
  - 대응: cut-off 정책 단순화 + 클라이언트 안내 문구 통일.

---

## 6) 이번 주 바로 시작할 P0 태스크
1. DLQ 테이블/권한/환경변수 반영.
2. 스케줄 발송 Lambda 재시도 상태머신 반영.
3. 실패 코드 표준 enum 정리(백엔드 공용).
4. Admin Docs에 dead-letter 조회/재처리 명령 추가.
5. 장애 드릴 시나리오 2종(전체 실패/부분 실패) 작성.

> 위 5개 완료 시, Phase 2의 “라이브 가능성”을 판단할 수 있는 최소 운영 루프가 만들어진다.

## 7) 베타 런칭 우선 전략 (결정사항)
- **즉시 개발 진행**: P0-1(관측 알람/대시보드) + P1-4(`/cheer/scheduled` 페이징 표준화)
- **문서 적재 후 베타 중 진행**:
  - P0-2 장애 드릴 2회 수행 계획/체크리스트
  - P0-3 Go-Live Gate 증적 템플릿(기능/신뢰성/관측/운영/롤백)
  - P1-5 성능 리허설 계획(p95/에러율 측정 시나리오)
- 운영 전략: 빠른 서버 테스트 후 베타 런칭, 베타 기간에 드릴/게이트 증적/성능 리허설을 병행한다.
