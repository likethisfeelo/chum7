# CHME 챌린지·퀘스트 정교화 실행 계획 (Admin-First / Live Service 지향)

## 1) 추진 원칙

- **Admin-First**: 참여자 앱은 어드민 운영 플로우가 안정화된 뒤 착수.
- **Live-Service 품질 우선**: MVP 축소 대신 운영/모니터링/장애대응까지 포함.
- **정책 코드화 우선**: Remedy, Layer B 승인, Layer D 공개 규칙을 화면보다 서버 도메인 규칙으로 먼저 고정.
- **릴리즈 게이팅**: 각 단계마다 “운영 가능 기준(Go-Live Gate)” 통과 시 다음 단계로 진행.

---

## 2) 전체 단계 계획 (권장)

## Phase 0. 기획 잠금 + 도메인 계약 (1주)

### 목표
- 현재 명세서의 정책을 **개발 계약서(ADR + API + 상태전이표)** 로 고정.

### 작업
- 챌린지 유형(`leader_only`, `personal_only`, `leader_personal`, `mixed`)별 허용 액션 매트릭스 작성.
- Layer A/B/D + Remedy 정책의 상태전이 다이어그램 확정.
- 에러 코드/HTTP 코드 표를 OpenAPI 스펙으로 고정.
- 시간 검증(`practiceAt/uploadAt/certDate`) 기준 명시:
  - 기준 타임존: **KST(Asia/Seoul)**
  - 저장 기준: 서버 영속 데이터는 UTC
  - Day 계산: 글로벌 확장을 고려해 **사용자 로컬 타임존의 00:00~23:59:59** 기준일 지원

### 산출물
- `OpenAPI` 초안
- `State Transition Table`
- `Policy Decision Log(ADR)`

### Go-Live Gate
- PM/개발/QA/운영이 동의한 단일 API 계약 문서 존재.

---

## Phase 1. 데이터 모델 + 백엔드 코어 (2~3주)

### 목표
- UI 없이도 정책이 동작하는 서버 코어 완성.

### 작업
- 스키마 반영: `Challenges`, `Quests`, `Verifications` 확장.
- 핵심 도메인 서비스 구현:
  - Quest Assignment Engine (A/B/D 활성 퀘스트 계산)
  - Remedy Validator
  - Verification Timestamp Validator
  - Score Engine (일반/Remedy/Extra)
- 중복 인증 `isExtra` 처리 및 `primaryVerificationId` 연결.
- Layer D `revealAt`/`previewNotifiedAt` 스케줄 처리.
- 완주 판정 엔진 구현:
  - Layer A/B: Remedy 포함 100% 달성 시 완주
  - Layer D: 적용 기간 내 미수행 항목은 Remedy 불가, 미완료 시 완주 실패
- 승인 운영 엔진 구현:
  - 기본 자동승인 + 관리자 수동승인 모드 전환 가능
  - 수동승인 모드에서 관리자 지정 보조승인자(도와주는 사람) 승인 가능
  - 승인 결과 정정: 승인/반려/재승인을 **당일 내** 변경 가능

### 테스트
- 정책 조합별 단위 테스트(Strict/Limited/Open × A/B/D).
- 상태전이 테스트(pending→approved/rejected/revision_pending/expired).
- 경계값 테스트(Day 6, endDay 만료, ±1 day mismatch, 4시간 룰).
- 로컬 타임존 Day 경계 테스트(UTC 저장값과 사용자 기준일 일치성).
- 승인 모드 전환/권한/당일 정정 상태전이 테스트.

### Go-Live Gate
- 핵심 도메인 테스트 커버리지 목표 달성(권장: 정책 모듈 85%+).

---

## Phase 2. 어드민 시스템 1차 (운영 필수 기능) (2주)

### 목표
- 리더/운영자가 실서비스 운영 가능한 최소 운영 콘솔 확보.

### 작업
- 챌린지 생성/수정(유형, remedyPolicy, personalQuest 옵션).
- Layer A/D 퀘스트 등록, 공개 시점 관리.
- Layer B 승인 큐:
  - 승인/반려(피드백 필수, 최소 10자)
  - 재심사 상태 필터
  - 마감 임박/만료 대상 뷰
- 승인정책 운영 UI:
  - 자동승인 ↔ 수동승인 전환
  - 보조승인자 지정/해제
  - 당일 승인 취소·반려·재승인 기능
- 운영 로그 뷰(누가 어떤 승인/반려/수정/정정을 수행했는지).
- 리더 퀘스트 수정 정책 반영:
  - 진행 중 내용 변경 금지
  - 오타/표현 정정만 허용(점수/검증방식/기간 변경 불가)

### Go-Live Gate
- 실제 운영자가 “챌린지 1개를 생성→운영→종료” 전 과정을 콘솔만으로 수행 가능.

---

## Phase 3. 어드민 시스템 2차 (운영 안정화) (1~2주)

### 목표
- 장애 대응/모니터링/품질 관리 기능 강화.

### 작업
- 대시보드:
  - 일별 인증율, 실패일 분포, Remedy 사용률
  - Layer B 승인 지표(평균 승인시간, 반려율)
  - 승인 정정 지표(당일 취소/재승인 건수)
- 알림 운영 기능:
  - Layer D 예고/공개 푸시 발송 이력
  - 재발송/실패 재처리 큐
- 감사/보안:
  - 관리자 액션 감사로그(승인 모드 변경/보조승인자 변경/정정 이력 포함)
  - 역할 기반 접근 제어(RBAC) 세분화

### Go-Live Gate
- 운영 이슈(알림 실패, 승인 지연, 정책 문의, 승인 실수)에 대해 콘솔에서 원인 파악 가능.

---

## Phase 4. 참여자 프론트엔드 (UX 검증 반영) (2~3주)

### 목표
- 어드민 코어를 기반으로 참여자 UX를 안정적으로 구현.

### 작업
- 오늘의 퀘스트/개인 퀘스트 표시 및 상태 안내.
- 인증 제출(일반/추가/Remedy) UX 분기.
- Day 6 Remedy 가이드 UI.
- 피드에서 `isExtra` 노출 정책:
  - 기본 나만보기
  - 챌린지 기간 내 공개 전환 가능
- `open + allowBulk=true` UX:
  - 여러 날 복구 가능 안내
  - 날짜별 소감 입력 필수 체크리스트 강제
- Layer D 예고 푸시 문구는 “곧 준비된 새 퀘스트가 열려요” 수준으로 운영.

### Go-Live Gate
- 시나리오 테스트(신규 참여자/실패 후 복구/개인 퀘스트 반려 후 재제출) 통과.

---

## Phase 5. 사전 출시 검증 (1주)

### 목표
- 실서비스 수준의 안정성 확보.

### 작업
- E2E 회귀 테스트(핵심 사용자 여정).
- 부하 테스트(인증 몰림 시간대, Day 6 Remedy 트래픽).
- 운영 런북 작성(장애, 수동 보정, 알림 장애 대응).

### Go-Live Gate
- 릴리즈 체크리스트 100% 통과 + 롤백 플랜 검증 완료.

## 3) 결정사항 반영본 (사용자 확정안)

1. **타임존 기준**: KST(Asia/Seoul) 고정 적용.
2. **Day 계산 기준**: 서버 저장은 UTC, Day 판단은 사용자 로컬 타임존의 00:00~23:59:59 기준.
3. **Layer D 예고 푸시**: “있다 준비해라” 수준의 가벼운 사전 안내로 공개.
4. **완주 조건**:
   - Layer A/B는 Remedy 포함 100%면 완주 가능.
   - Layer D는 해당 기간 미참여 항목 Remedy 불가, 해당 미완료가 있으면 완주 불가.
5. **리더 퀘스트 수정**: 진행 중 수정 금지, 오타 수정만 허용.
6. **Layer B 반려 피드백 정책**: 최소 10자 + 품질 필터 적용.
7. **open + allowBulk UX**: 날짜별 소감 입력 필수 가이드 유지.
8. **추가 인증 공개 전환**: 챌린지 기간 내 공개 전환 가능.
9. **RBAC 단위**: 현재 구조와 충돌 없음. 중복/세부역할은 후속 정의.
10. **운영 SLA 수치화**: 추후 운영팀과 별도 확정(현재 단계에서는 수치 목표 미고정).
11. **승인 운영 정책(초기 포함)**:
    - 자동승인 기본, 필요 시 관리자 수동승인 전환 가능
    - 수동승인 시 관리자 지정 보조승인자 승인 가능
    - 이미 승인된 건도 당일 내 취소/반려/재승인 가능
    - 모든 변경은 감사로그에 남겨 운영 실수 대응 가능

## 4) 우선순위 백로그 (Admin-First)

### P0 (반드시 먼저)
- Remedy 검증 엔진
- Layer B 승인 상태머신
- Layer D 공개 스케줄러
- 관리자 승인 큐 + 반려 피드백
- 승인 모드 전환 + 보조승인자 + 당일 정정 + 감사로그

### P1 (출시 전 필수)
- 운영 대시보드(승인/실패/복구/정정)
- 알림 이력/재처리
- 감사로그 + RBAC

### P2 (출시 후 개선)
- RBAC 세부 역할/중복 권한 정리
- 추가 인증 공개 정책 A/B 테스트
- 난이도 추천 자동화

---

## 5) 리스크와 선제 대응

- **정책 복잡도 증가 리스크**: Remedy/Layer 조합 폭발
  → 대응: 정책 엔진 모듈화 + 조합 테스트 자동화.

- **운영자 승인 병목/실수 리스크**: Layer B 검토량 급증 + 승인 정정 필요
  → 대응: 자동승인 기본, 관리자 수동승인 전환, 보조승인자 운영, 당일 정정(취소/반려/재승인), 전 과정 감사로그 필수.

- **시간 관련 분쟁 리스크**: practiceAt 조작 의심
  → 대응: 4시간 룰 고정 + 에러 메시지 상세화 + 운영 로그 제공.
