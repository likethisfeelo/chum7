# 챌린지 인증유형 확장 점검 및 작업계획

## 1) 현황 점검 요약

### A. ME/챌린지 피드 인증 작성 UI
- 현재 인증 작성은 `InlineVerificationForm` 기준으로 `todayNote`(텍스트) + (선택) 미디어 첨부 방식입니다.
- UI에서 첨부 가능한 타입은 `image/*`, `video/*`만 열어두고 있으며, 링크 입력 필드는 없습니다.
- 즉, UX 관점에서는 사실상 **텍스트 단독 또는 텍스트+미디어(이미지/영상)** 구조이고, “링크 인증” 전용 입력 플로우는 미구현 상태입니다.

### B. 업로드/제출 백엔드
- 업로드 URL 발급 API는 이미지/영상 MIME 타입을 허용합니다.
- 하지만 실제 인증 저장 API(`POST /verifications`) 입력 스키마는 `imageUrl`만 별도 필드로 받으며,
  `videoUrl`/`linkUrl`/`verificationType` 같은 타입 구분 필드가 없습니다.
- 따라서 현재 구조는 영상 파일 업로드 자체는 가능해도, 데이터 모델 상 “영상 인증”을 명시적으로 관리하기 어렵습니다.

### C. 챌린지 생성(어드민) 인증유형 설정
- 백엔드 생성 스키마의 `allowedVerificationTypes`는 이미 `['image','text','link','video']`의 **배열**이며 기본값도 전체 허용입니다.
- 즉 API 레벨은 “기본 전체 허용 + 선택적 제한” 요구사항과 거의 일치합니다.
- 다만 실제 어드민 UI가 이 배열을 체크박스/멀티셀렉트로 보내는지(혹은 단일 선택으로 축약하는지)는 화면 구현 위치별 점검이 필요합니다.

---

## 2) 요구사항 반영 목표 정의

1. ME 인증 작성에서 다음 유형을 모두 지원
   - 텍스트
   - 사진
   - 60초 이내 영상
   - 링크
2. 챌린지 생성 시 기본값은 모든 인증유형 허용
3. 챌린지 생성자가 특정 유형만 선택해서 제한 가능(멀티 선택)
4. 챌린지별 허용 타입에 따라 인증 작성 UI/검증이 일관되게 동작

---

## 3) 작업계획 (실행 순서)

## Phase 1. 데이터 계약(Contract) 정리
1. `VerificationContent` 표준 스키마 도입
   - 예시
     - `verificationType: 'text' | 'image' | 'video' | 'link'`
     - `content: { text?: string; imageUrl?: string; videoUrl?: string; linkUrl?: string }`
2. 기존 `todayNote` 중심 모델과의 하위호환 전략 정의
   - 마이그레이션 전환기간 동안 구필드/신필드 병행 파싱
3. API 에러 코드 표준화
   - `VIDEO_DURATION_EXCEEDED`
   - `UNSUPPORTED_VERIFICATION_TYPE`
   - `LINK_URL_INVALID`

## Phase 2. ME 인증 작성 UX 개편
1. 인증유형 탭/토글 추가
   - 텍스트 / 사진 / 영상 / 링크
   - `allowedVerificationTypes` 기반으로 노출 제어
2. 영상 60초 제한
   - 클라이언트: 파일 선택 후 메타데이터(`duration`) 검사
   - 서버: 최종 방어 로직(메타데이터 검증 또는 후처리 검증)
3. 링크 입력 UX
   - URL 포맷 검증
   - 미리보기(도메인/제목) 옵션은 2차 적용
4. 제출 페이로드 분기
   - 타입별로 필요한 필드만 전송

## Phase 3. 백엔드 검증/저장 확장
1. `POST /verifications` 스키마 확장
   - 타입별 required/optional 규칙 적용
2. 저장 모델 확장
   - `verificationType`, `videoUrl`, `linkUrl` 등 저장
3. 기존 조회 API 응답 일관화
   - 피드/내기록/상세 조회에서 동일한 타입 렌더 가능하도록 응답 정규화
4. 보안/정책
   - 링크 도메인 allow/deny 정책(필요 시)
   - 영상 MIME/용량/길이 제한

## Phase 4. 어드민 챌린지 생성 UX 보강
1. 인증유형 선택 UI를 멀티 선택(체크박스)으로 고정
2. 초기값: 전체 선택
3. 최소 1개 선택 validation
4. 수정(Edit) 화면에서도 동일 규칙 유지

## Phase 5. QA 시나리오
1. 허용 타입 조합별 작성 가능 여부
   - 예: `['text']`, `['video']`, `['link','image']`, 전체
2. 영상 60초 경계값 테스트
   - 59초(성공), 60초(정책 합의 필요), 61초(실패)
3. 링크 유효성 테스트
   - http/https, 비정상 URL, 매우 긴 URL
4. 회귀 테스트
   - 기존 텍스트+사진 흐름, 추가기록/보완인증 영향 확인

---

## 4) 예상 리스크 및 선결정 필요사항

1. **영상 길이 60초 기준값 포함 여부**
   - “60초 이내”를 `<= 60`으로 볼지 `< 60`으로 볼지 확정 필요
2. **텍스트 필수 여부**
   - 영상/링크 인증에서도 `todayNote`를 필수로 유지할지 정책 결정 필요
3. **기존 데이터 호환**
   - 구 스키마 레코드 렌더링 fallback 우선 적용 필요
4. **어드민 UI 소스 오브 트루스 위치 정리**
   - 실제 운영 중인 어드민 프론트 경로를 확정하고 해당 코드만 수정해야 중복 개발 방지 가능

---

## 5) 바로 착수 가능한 구현 단위(권장)

1. 계약 정의(PR-1): 타입/스키마/응답 포맷 정리
2. 프론트 작성폼(PR-2): 타입 토글 + 링크 입력 + 영상 길이 검사
3. 백엔드 제출(PR-3): 타입별 validation + 저장 필드 확장
4. 어드민 생성폼(PR-4): 인증유형 멀티 선택 + 기본 전체 선택
5. QA/회귀(PR-5): e2e 및 경계 테스트

위 순서로 가면, 현재 서비스 영향(회귀)을 최소화하면서 요구사항을 단계적으로 반영할 수 있습니다.
